Frequently Asked Questions (FAQ)

 

1.      Question: The presentation said that the agents carry with them some "state" information. Does this mean that you could partially execute at a given host, and then continue elsewhere?

 

Answer: Yes, an agent can “partially execute at a given host, and then continue elsewhere”. Think of the agent as a software object. By saying that its state is saved is meant that values of its fields and properties would be saved between transfers.

 

2.      Question: Is it possible that the host requiring a job to be done can have parts of his job executed at other hosts on the network too?

 

Answer: Yes, it is possible. In fact, as we said in the presentation, mobile agents can be used for parallel processing. You can divide your big computation into smaller sub-computations, put these sub-computations in agents, and dispatch your agents to different hosts on the network. You can thus use the CPU cycles of the other hosts for your computation.

 

3.      Question: Who's going to decide where the mobile agents are at a given time? If the situation can be dynamic, who's going to do the book-keeping, and how does a host find out where the agent of interest is currently available? Have protocols been developed for these? What are they called?

 

Answer: Mobile agents are autonomous by definition. So a mobile agent can choose a host where it wants to execute on. This leads to the question -- how to locate an agent? No support has been provided in the EtherYatri Framework for locating agents but this can easily be implemented by the users themselves. You could, for example, have a well-known agent at a well-known host. Let’s call this agent – RegistryAgent. All other agents register their current locations with RegistryAgent. So to locate an agent of interest you contact RegistryAgent and ask it for a handle to the agent of your interest.

 

4.      Question: My major concern here is that you'll probably need some kind of a distributed manager, to regulate the whole stuff. Am I on the wrong track?

 

Answer: You are on the right track. RegistryAgent is probably what you have in mind.

 

5.      Question: The biggest thing with client-server, was control and hence security. Security is a really big thing these days, and I'm inclined to think that the agents are typically less secure and more prone to security attacks. What do you think?

 

Answer: There are issues with mobile code in general. But EtherYatri runs on top of the .NET Framework and hence can use the powerful .NET security mechanisms to make sure that no agent can do harm to your machine and no machine can do harm to your agent.

 

6.      Question: Do you have some samples which demonstrate the writing of mobile agents using EtherYatri.NET?

 

Answer: Check out the EtherYatri Homepage. In the downloads section you will find links to a demo and some mobile agent samples.
